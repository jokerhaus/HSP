# HSP v0.1.0

Release date: 2026-04-23

This is the first HSP release. It packages the secure storage core, native HSP
runtime, HTTP/3 gateway, S3-like service, CDN-like service, Go SDK, CLI,
conformance harness, and production-hardening documentation.

## Product Summary

HSP is a secure object storage protocol for applications that need more than a
basic file bucket. It is useful when you want:

- encrypted object storage by default
- tenant isolation across data, cache, logs, metrics, and policies
- S3-like integration for existing tools
- CDN-like ciphertext delivery for hot downloads
- cheap existence and integrity checks without downloading the object
- storage-layer metrics and structured logs for operations teams

## Example Use Cases

- A media backend stores encrypted photos and videos, then verifies uploads by
  CID and size without downloading the object again.
- A SaaS platform gives every customer isolated namespaces and object keys.
- A migration tool moves objects into HSP through an S3-like endpoint while the
  core service still enforces HSP auth and policy rules.
- An edge layer serves immutable CID URLs with long cache lifetimes and mutable
  bucket/key URLs with short TTL and event-based purge.

## Highlights

- Native HSP protocol contracts and Rust runtime.
- HTTP/3 gateway for HSP operations.
- S3-like bucket/object service through `hsp-s3`.
- CDN-like edge distribution through `hsp-cdn`.
- Public profile remains ciphertext-only.
- `HEAD` and metadata endpoints expose:
  `exists`, `deleted`, `size_bytes`, `ciphertext_size_bytes`,
  `integrity_hash`, `cid`, `created_at_ms`, `encryption_profile_id`, and
  `key_policy_id`.
- Prometheus metrics at `/metrics`.
- Structured secret-aware logs at `/v1/observability/logs`.
- Go SDK and CLI support for the alpha/beta surfaces.
- Black-box conformance across native, gateway, S3-like, and CDN-like paths.

## Security Notes

- HSP public profile does not serve plaintext object data.
- Server and edge layers work with ciphertext unless a separate trusted-edge
  profile is explicitly enabled.
- All persisted stores are designed for envelope encryption.
- Mutation operations use replay protection and capability checks.
- Namespace mutations are signed.
- Chunk upload now rejects data when the declared chunk CID does not match the
  received ciphertext bytes.

## Known Production Sign-Off Boundary

The codebase and release gates are ready for the first release. Strict
production approval still requires independent external security review and
operator-attested staging or production drills for key recovery, incident
response, replication lag, and CDN purge failure recovery.
