# Upgrade Guide: HSP v0.1.1

This release upgrades `v0.1.0` with security hardening and release evidence.
The public HSP, gateway, S3-like, and CDN-like API surfaces remain compatible.

## Required Operator Changes

### Set Runtime Secrets Explicitly

Every runtime environment must provide strong, unique values:

```bash
export HSP_KMS_SEED="$(openssl rand -base64 48)"
export HSP_EDGE_SIGNING_SECRET="$(openssl rand -base64 48)"
```

Do not reuse examples, test literals, or old defaults such as
`hsp-secure-alpha-local-seed`, `hsp-s3-runtime-seed`, `hsp-cdn-runtime-seed`, or
`edge-secret`; the services now reject them.

### Choose AWS KMS Runtime Mode Explicitly

For live AWS KMS usage:

```bash
export HSP_AWS_KMS_RUNTIME=live-cli
export HSP_AWS_KMS_KEY_ALIAS=alias/hsp-storage
export HSP_AWS_KMS_REGION=us-east-1
```

For local development only:

```bash
export HSP_AWS_KMS_RUNTIME=local-dev
```

The legacy implicit fallback mode is no longer accepted.

### Align Go Toolchain

Use Go `1.25.10`:

```bash
GOTOOLCHAIN=go1.25.10+auto go work sync
GOTOOLCHAIN=go1.25.10+auto go test ./sdk/go/... ./cli/hspctl/...
```

## Kubernetes Changes

The provided manifests now expect runtime secret values from the shared
`hsp-runtime-secrets` secret:

- `storage-kms-seed` for `HSP_KMS_SEED`
- `edge-signing-secret` for `HSP_EDGE_SIGNING_SECRET`

Services that share encrypted storage must share the same storage KMS key
domain. Use separate secrets only for isolated environments or storage domains.

## Security Behavior Changes

- Oversized native `DATA` frames are rejected before payload allocation.
- Replay markers are durable and encrypted, so replay protection survives
  process restart.
- S3/CDN scoped credentials cannot cross bucket/path boundaries.
- Scoped credentials cannot read direct CID objects unless the CID is proven to
  be bound to the authorized bucket/key.
- CDN cache keys are canonicalized and tenant-scoped.
- Edge tokens must match the request method.

## Recommended Validation

Run the full review bundle before deploying:

```bash
./scripts/generate_release_review_bundle.sh
```

At minimum, verify:

```bash
cargo test --workspace --all-targets
cargo clippy --workspace --all-targets -- -D warnings
env GIT_CONFIG_GLOBAL=/dev/null GIT_CONFIG_SYSTEM=/dev/null cargo audit
GOTOOLCHAIN=go1.25.10+auto go test ./sdk/go/... ./cli/hspctl/...
cd sdk/go && GOTOOLCHAIN=go1.25.10+auto "$(go env GOPATH)"/bin/govulncheck ./...
cd ../../cli/hspctl && GOTOOLCHAIN=go1.25.10+auto "$(go env GOPATH)"/bin/govulncheck ./...
cargo run -p hsp-conformance
```

## Rollback Notes

If rollback is required, keep encrypted-store backups and runtime secret
material together. Do not roll back to an environment that uses rejected default
secrets. If `v0.1.1` has written durable replay markers, preserving the same
encrypted store and KMS domain is the safest rollback path.
